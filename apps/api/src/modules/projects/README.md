# Projects

CRUD de proyectos, actualización de MDD y entregables (Blueprint, API, etc.).

- **Stage (Prisma):** `mddContent`, semáforo SDD (`status` ROJO/AMARILLO/VERDE), `precisionScore`, `estimation` y `workflowStatus` (DRAFT/ACTIVE/…) viven por etapa. `PATCH /projects/:id` puede llevar `stageId` para apuntar a una etapa concreta; si no, se usa la etapa activa (`workflowStatus === ACTIVE`) o la de menor `ordinal`.
- **Rutas etapas:** `GET /projects/:projectId/stages` (lista ordenada por `ordinal`, incluye `estimation`); `POST /projects/:projectId/stages` (respuesta **`{ stage }`**, no proyecto completo; opcional `copyMddFromStageId`, `activate`); `PATCH /projects/:projectId/stages/:stageId` (respuesta **`{ stage }`**; nombre/clave/ordinal, `activate: true` para exclusividad ACTIVE).
- **Contratos API:** `POST …/generate-api-contracts` y su **preview** exigen que el Blueprint cubra el **§3 Modelo de datos** del MDD (`ConformanceService.checkBlueprintDataModel`); si hay gaps → **400** con `code: BLUEPRINT_DATA_MODEL_GAPS` y `gaps[]`.
- **Conformance:** `GET …/conformance` incluye **`blueprintDataModel`** (heurístico, siempre presente); con `?useLlm=true` el resto puede venir del LLM pero **`blueprintDataModel` sigue siendo el chequeo determinista §3 vs Blueprint**. **`GET …/audit-documents`** — auditoría integral (conformidad + `conformanceSummary` + gaps SDD: inventario DBGA/§3, decision log BRD, trazabilidad entidad→API, integraciones WebSocket/Banxico/Polygon); expuesto también como MCP `audit_documents`.
- **ProjectsService:** fachada HTTP/MCP/orquestador — CRUD ligero (`findAll`, `findOne`, favorites, `remove`), delegación a servicios especializados. Utils: `project-api.util.ts` (`toApiProject`), `project-list-item.util.ts`.
- **ProjectLifecycleService:** alta y clonación (`create`, `cloneProject`) — grupo default, hooks de plugins, wire Ariadne legacy, sync FalkorDB en clon legacy.
- **ProjectEstimationRecalcService:** Upsert de `Estimation` por `stageId` (horas/MXN/team) cuando cambian MDD o infra.
- **ProjectStageService:** CRUD de etapas (`createStage`, `patchStage`, `transitionStage`, `activateStageExclusive`), listado (`listStages`), entregables por etapa (`getStageDeliverables`) y detalle Workshop (`getStageDetail`). Integración FalkorDB en etapas legacy al crear.
- **ProjectUxGuideService:** generación LLM (`generateUxUiGuide`), composición desde referencia (`composeUxGuideFromDesignRef`) y reparación YAML (`repairUxUiGuideYaml`).
- **ProjectDeliverableGeneratorsService:** generadores thin de entregables (`generateDocument`, `generateSpec`, `generateBlueprint`, `generateApiContracts`, `generateTasks`, `generateAem`, gobernanza de agentes, etc.). `ProjectsService` delega la API pública sin cambiar contratos HTTP/MCP.
- **ProjectDeliverableGateService:** gate MDD de entrega (`assertDeliverablesAllowed`, `assertMddDeliveryGateForDeliverables`) y recálculo de semáforo (`refreshStageSemaphoreFromProject` → métrica integral de `EstimationService`, misma fuente que el panel Workshop / listado). Utils: `project-semaphore.util.ts`.
- **ProjectConformanceService:** auditoría y conformidad (`auditDocuments`, `getConformance`, `verifyDeliverable`). Para **LOW/MEDIUM** usa gaps de `low-medium-readiness.util.ts` (HU↔Tasks, Spec↔API↔UX) en lugar de conformidad MDD↔derivados; semáforo compuesto vía `project-semaphore-composite.util.ts` con trazabilidad BRD→Spec.
- **ProjectBrdService:** generación greenfield de BRD desde DBGA (`suggestBrdFromDbga`).
- **ProjectUpdateService:** `PATCH /projects/:id` (`update`), snapshots DBGA/Spec (`listDocumentSnapshots`, `restoreDocumentSnapshot`), pipeline MDD en patch, recálculo de estimación y semáforo.
- **ProjectComplexityService:** benchmark Paso 0 (`generateBenchmark`), re-valoración HITL (`reassessComplexity`), confirmación/rechazo de complejidad (`confirmComplexityProposal`, `tryConfirmComplexityFromChatMessage`).
- **ProjectPhase0Service:** Deep Research Fase 0 (`phase0DeepResearch`) — scraping + resumen LLM persistido en `phase0SummaryContent`.
- **ProjectSddReconcileService:** post-regeneración SDD (`runPostRegenSddConflictSurfacing`, `runPostRegenSddAutoReconcile` deprecated).
- **DeliverablesCascadeService:** orquestación multi-ola (`generateDeliverablesCascade`), sync pantallas MCP (`syncUiScreens`), post-pase W4 y retry de conformidad API/Infra; convergencia post-cascada con routing por `complexity` (HIGH: MDD repair + gaps; LOW/MEDIUM: regen dirigida HU/Spec/API/Tasks). **`repairReadinessGaps`** — misma convergencia bajo demanda (`POST /projects/:id/repair-sdd-gaps`, job `repair-sdd-gaps`) desde el Semáforo (Brechas SDD). Utils: `deliverables-cascade-gaps.util.ts` (feedback inline por oleada según complejidad), `constitution-markdown.util.ts`, `sync-domain-inventory-stage.util.ts`.
- **ProjectMddPersistService:** persistencia MDD desde jobs en background (`persistMddFromBackgroundJob`), PATCH (`persistMddFromPatch`), snapshot de delivery gate y errores estructurados del pipeline (`throwMddPipelineBadRequest`). Utils: `project-mdd-persist.util.ts` (`buildSemaphoreBaseFromProject`, `mergeProjectFieldsForSemaphore`, `resolveMddPersistMode`), `project-access.util.ts` (`loadAccessibleProjectWithStages`).
- **stage-helpers.ts:** `pickPrimaryStage`, `flattenStageDeliverables`.
- **Engine (MddUpdatePipelineService):** Validación de MDD al persistir en etapa. Antes del delivery gate, `applyPreDeliveryGateFixes` ejecuta reparación determinista (calidad §1/§3/§4/§7, coherencia cruzada §2–§7, Mermaid, stub `TechnicalMetadata`); defectos auto-reparables pasan a **warnings** y disparan el **auto-loop** de agentes sin bloquear al usuario.
- **Patrones SSOT:** En `PATCH` con `mddContent`, salvo `allowGovernancePatternChange: true` (wizard), se restaura la sección `[ARQUITECTURA - SECCIÓN INMUTABLE]` según la selección `[X]` del último guardado (`enforceMddGovernancePatternsOnPersist`). Jobs MDD en background (`persistMddFromBackgroundJob`) anclan los IDs de patrones al inicio del job (editor + BD) y los reinyectan tras cada borrador y al finalizar, aunque `prepareMddForOutput` no los conserve. Respuesta puede incluir `mddGovernancePatternsReverted: true`. Con `allowGovernancePatternChange: true` se omite el pipeline pesado (solo `storeMddMarkdownForPersist`). `mddGovernanceSeedOnly: true` omite el pipeline al guardar solo la semilla SSOT. `clearMddCompletely: true` vacía el MDD sin reinyectar patrones. El MDD **no** lleva cabecera de fechas en markdown (solo cuerpo limpio).
- **Complejidad HITL:** `complexityPending` (JSON) propone nivel + plan hasta confirmación en el chat o vía `POST /projects/:id/confirm-complexity`. Con propuesta pendiente, `POST /projects/:id/generate-deliverables` responde 400.
- **Entregables:** `POST /projects/:id/generate-deliverables` itera oleadas `DELIVERABLE_WAVES_BY_COMPLEXITY` (HIGH: W1 spec+architecture → W2 UC/HU/API/flujos/UX/blueprint → W2b pantallas → W3 tasks/infra/gobernanza → **W4 post-pase**). Tras W3 persiste **bundle atómico** US+pantallas+API+tasks (`deliverable-bundle-persist.util.ts`, `bundleVersion` en snapshot). W4 incluye trazabilidad cross-artifact (pantalla→US→API→task), cardinalidad MDD §3 y reintento tasks. **US estables:** `US-CRUD-*` / `US-JRN-*` (`us-id-registry.ts`). **Pantallas v1:** solo rutas con API o HU journey; resto → «Fuera de alcance v1» (`ui-screens-v1-scope.util.ts`). **Tasks SSOT:** `tasksJson` v2 + fallback `tasksContent` (`tasks-resolve.ts`, spec-kit export).
- **Jobs en background (`DeliverablesQueueService`):** cola BullMQ `theforge-deliverables` (o in-memory sin Redis). Estado: `GET /projects/:id/deliverables-jobs/:jobId`, `GET /projects/jobs/:jobId`, `GET /projects/:id/generation-status`. **Cancelación:** `DELETE /projects/:id/deliverables-jobs/:jobId` o `DELETE /projects/jobs/:jobId?projectId=` — jobs en cola se eliminan al instante; jobs activos reciben abort cooperativa (flag Redis + `AbortSignal` entre oleadas/post-pase/convergencia). Mientras el worker cierra el paso LLM en curso, `generation-status.busy` deja de incluir ese job (banner se limpia) pero los gates pueden seguir bloqueando un nuevo enqueue hasta que el abort termine. Cancelación de usuario → `UnrecoverableError` (sin reintentos BullMQ).
- **Integración NEW ↔ LEGACY:** `ProjectIntegrationController` bajo `/projects/:id/integration/*` — enlace bidireccional, handoff `NEW-LEG-*`, import en etapa 2+, **`POST …/integration/promote-to-stage`** (nueva etapa desde SENT), trazabilidad `IntegrationTrace`. **`GET …/stages/:stageId/deliverables`** — snapshot o live (P0.1e). **IntegrationAgent:** **`POST …/integration/sync-handoff-spec`** (y `.../stages/:stageId/...`) (re)genera `handoff-spec.md` desde los items NEW-LEG (`IntegrationAgentService`, entregable `handoffSpecContent`, pestaña Workshop *Handoff Spec*). Ver `integration/README.md` y `docs/plans/PLAN-LEGACY-NEW-INTEGRATION.md`.
- **Fusión Paso 0:** `POST /projects/merge` — fusiona 2+ proyectos del mismo owner en DBGA/borrador Fase 0 (`project-merge.service.ts`). Depende de `Phase0Module` (no `AiAnalysisModule` completo) para evitar dependencia circular Nest. Body: `projectMergeBodySchema` (`@theforge/shared-types`).
- **Renombrar:** `PATCH /projects/:id` con `{ name }` — solo el propietario (`userId`); ya en `updateProjectSchema`.
- **Clonar:** `POST /projects/:id/clone` con body opcional `{ name?, visibility? }` — copia documentos de `Project`, todas las `Stage` (MDD, BRD, entregables, snapshots, estimación) y metadatos SDD; **no** copia sesiones/chat, favoritos, webhooks converge, enlaces NEW↔LEGACY ni trazas. El clon es `PRIVATE` del usuario actual por defecto. Respuesta incluye `clonedFromProjectId`. Util: `project-clone.util.ts`.
- **Portabilidad Notion (proyecto completo):** `GET /projects/:id/export/notion` → ZIP Markdown & CSV (convención Notion: páginas `.md`, carpetas por etapa, CSV handoff/trazas/etapas, `index.html`, `_theforge/manifest.json`). `POST /projects/import/notion` (multipart `file`) restaura proyecto standalone. `POST /projects/import/notion/pair` (`newProject` + `legacyProject` ZIP) restaura vínculo NEW↔LEGACY y `IntegrationTrace`. Utils: `project-notion-export.util.ts`, `project-notion-import.util.ts`, `project-notion-portability.service.ts`; tipos `@theforge/shared-types` (`project-notion-portability.ts`).
- **`PATCH specContent`:** validado con `validateDocumentForPersist` — rechaza vaciar Spec con contenido sustancial, shell solo-changelog o reducción peligrosa (>45% del cuerpo previo).
- **`PATCH dbgaContent`:** misma validación que Spec (incl. solo-changelog y reducción peligrosa). Antes de sobrescribir guarda **snapshot** completo (`DocumentSnapshot`, máx. 25 por campo). Cabecera `Creado` / `Última regeneración` vía `stampMarkdownIfBodyChanged` — no re-stampa si el cuerpo no cambió (evita bucles de auto-guardado en Workshop). **`GET …/document-snapshots?field=dbgaContent`** y **`POST …/document-snapshots/:id/restore`** para recuperación manual. Ver `document-snapshot/`.
- **SDD / spec-kit:** `GET /projects/:id/export/sdd-bundle` (JSON layout spec-kit); `GET /projects/:id/export/repo-handoff` (spec-kit + gobernanza reconciliada o sintetizada + espejos `docs/sdd/` + `openspec/BRANCH-POLICY.md`); `POST /projects/:id/clarify-spec` (legacy Spec); **`POST /projects/:id/clarify-document`** (cualquier entregable, marca `[NEEDS CLARIFICATION]`); **`POST /projects/:id/resolve-clarifications`** (respuestas del usuario → documento regenerado); `GET /projects/:id/analyze?stageId=`; `GET /projects/:id/next-task`; `POST /projects/:id/converge?stageId=`; `POST /projects/:id/tasks-to-issues` (`GITHUB_TOKEN`). Ver `sdd-integration.service.ts`, `document-clarify.util.ts`, `handoff-export.util.ts`, `docs/plans/PLAN-SPECKIT-ALIGNMENT.md` y `docs/speckit-vs-theforge.md`.
- **Re-valorar complejidad:** `POST /projects/:id/reassess-complexity` (body opcional `{ note?: string }`) — vuelve a inferir `complexityPending` desde DBGA/MDD/Spec existentes sin re-ejecutar el stream DBGA.
- **BRD greenfield:** `POST /projects/:id/suggest-brd-from-dbga` (body opcional `{ stageId?: string }`) — borrador BRD desde `Project.dbgaContent` (mín. ~300 caracteres); rechaza `projectType === LEGACY` (usar `POST …/legacy/suggest-brd-from-codebase-doc`). Prompt `brd-generation-prompt`: negocio puro (capacidades, UAT, entidades, políticas comerciales); **§4 diagramas Mermaid** (ecosistema, ER de negocio, 2–3 flujos); traduce lo técnico del DBGA; sin endpoints ni SQL.
- **AEM:** `POST /projects/:id/generate-aem` (body `{ marketScope: "global" \| "mexico" \| "latam" }`) — Análisis y Estudio de Mercado desde Benchmark (Deep Research), Fase 0 (DBGA) y BRD; segundo paso añade **Dictamen de inversión digital** (SEGUIR / NO SEGUIR / SEGUIR CON CONDICIONES). Persistido en `aemContent`. Prompts `aem-prompt.md` + `aem-investment-advisor-prompt.md`.
