# Projects

CRUD de proyectos, actualización de MDD y entregables (Blueprint, API, etc.).

- **Stage (Prisma):** `mddContent`, semáforo SDD (`status` ROJO/AMARILLO/VERDE), `precisionScore`, `estimation` y `workflowStatus` (DRAFT/ACTIVE/…) viven por etapa. `PATCH /projects/:id` puede llevar `stageId` para apuntar a una etapa concreta; si no, se usa la etapa activa (`workflowStatus === ACTIVE`) o la de menor `ordinal`.
- **Rutas etapas:** `GET /projects/:projectId/stages` (lista ordenada por `ordinal`, incluye `estimation`); `POST /projects/:projectId/stages` (respuesta **`{ stage }`**, no proyecto completo; opcional `copyMddFromStageId`, `activate`); `PATCH /projects/:projectId/stages/:stageId` (respuesta **`{ stage }`**; nombre/clave/ordinal, `activate: true` para exclusividad ACTIVE).
- **Contratos API:** `POST …/generate-api-contracts` y su **preview** exigen que el Blueprint cubra el **§3 Modelo de datos** del MDD (`ConformanceService.checkBlueprintDataModel`); si hay gaps → **400** con `code: BLUEPRINT_DATA_MODEL_GAPS` y `gaps[]`.
- **Conformance:** `GET …/conformance` incluye **`blueprintDataModel`** (heurístico, siempre presente); con `?useLlm=true` el resto puede venir del LLM pero **`blueprintDataModel` sigue siendo el chequeo determinista §3 vs Blueprint**.
- **ProjectsService:** `GET /projects` (`findAll`) devuelve **listado ligero** (`project-list-item.util.ts`): metadatos, semáforo y etapas sin markdown ni entregables. `GET /projects/:id` (`findOne`) carga el proyecto **completo** con documentos. Ambos aplanan `mddContent`, `status`, `precisionScore` y `estimation` desde la etapa principal para compatibilidad con el front.
- **ProjectEstimationRecalcService:** Upsert de `Estimation` por `stageId` (horas/MXN/team) cuando cambian MDD o infra.
- **stage-helpers.ts:** `pickPrimaryStage`, `flattenStageDeliverables`.
- **Engine (MddUpdatePipelineService):** Validación de MDD al persistir en etapa.
- **Patrones SSOT:** En `PATCH` con `mddContent`, salvo `allowGovernancePatternChange: true` (wizard), se restaura la sección `[ARQUITECTURA - SECCIÓN INMUTABLE]` según la selección `[X]` del último guardado (`enforceMddGovernancePatternsOnPersist`). Respuesta puede incluir `mddGovernancePatternsReverted: true`. `mddGovernanceSeedOnly: true` omite el pipeline al guardar solo la semilla SSOT. `clearMddCompletely: true` vacía el MDD sin reinyectar patrones.
- **Complejidad HITL:** `complexityPending` (JSON) propone nivel + plan hasta confirmación en el chat o vía `POST /projects/:id/confirm-complexity`. Con propuesta pendiente, `POST /projects/:id/generate-deliverables` responde 400.
- **Entregables:** `POST /projects/:id/generate-deliverables` itera oleadas `DELIVERABLE_WAVES_BY_COMPLEXITY` (HIGH: W1 spec+architecture → W2 UC/HU/API/flujos/UX/blueprint → W2b pantallas → W3 tasks/infra/gobernanza → **W4 post-pase**). W4 regenera solo artefactos con gaps de precisión: architecture + logic-flows + api **en paralelo**, tasks **al final** (sin reintentos inline en W1/W3). **Tasks con coordenadas exactas:** al generar `tasks.md`, si hay `theforgeProjectId`, ChangeScope en la etapa, navigation map Ariadne o árbol `modules/` en architecture, se activa modo coordenadas (Archivo, Función, Línea, diff sugerido) vía `tasks-coordinates-context.util.ts` + `ResolveChangeToFilesService`. Rechaza `projectType === LEGACY` (usar `POST …/legacy/generate-deliverables`). **Cola background:** `POST …/generate-*` encola por defecto (`?queue=false` solo para sync explícito). `ProjectGenerationGuardService` + `project-generation-guard.ts` (shared-types): **un job activo por proyecto**, gates por oleadas (upstream persistido; job en cola no cuenta como listo), bloqueo durante stream MDD. `GET /projects/:id/generation-status` expone `{ busy, activeJob, gates }`. **BullMQ:** con `REDIS_URL`, jobs persistentes (`theforge-deliverables`); progreso vía `GET /projects/:id/deliverables-jobs/:jobId` o SSE. Sin Redis: cola in-memory **secuencial por proyecto** en el proceso API.
- **Integración NEW ↔ LEGACY:** `ProjectIntegrationController` bajo `/projects/:id/integration/*` — enlace bidireccional, handoff `NEW-LEG-*`, import en etapa 2+, **`POST …/integration/promote-to-stage`** (nueva etapa desde SENT), trazabilidad `IntegrationTrace`. **`GET …/stages/:stageId/deliverables`** — snapshot o live (P0.1e). **IntegrationAgent:** **`POST …/integration/sync-handoff-spec`** (y `.../stages/:stageId/...`) (re)genera `handoff-spec.md` desde los items NEW-LEG (`IntegrationAgentService`, entregable `handoffSpecContent`, pestaña Workshop *Handoff Spec*). Ver `integration/README.md` y `docs/plans/PLAN-LEGACY-NEW-INTEGRATION.md`.
- **Fusión Paso 0:** `POST /projects/merge` — fusiona 2+ proyectos del mismo owner en DBGA/borrador Fase 0 (`project-merge.service.ts`). Depende de `Phase0Module` (no `AiAnalysisModule` completo) para evitar dependencia circular Nest. Body: `projectMergeBodySchema` (`@theforge/shared-types`).
- **Renombrar:** `PATCH /projects/:id` con `{ name }` — solo el propietario (`userId`); ya en `updateProjectSchema`.
- **Clonar:** `POST /projects/:id/clone` con body opcional `{ name?, visibility? }` — copia documentos de `Project`, todas las `Stage` (MDD, BRD, entregables, snapshots, estimación) y metadatos SDD; **no** copia sesiones/chat, favoritos, webhooks converge, enlaces NEW↔LEGACY ni trazas. El clon es `PRIVATE` del usuario actual por defecto. Respuesta incluye `clonedFromProjectId`. Util: `project-clone.util.ts`.
- **`PATCH specContent`:** validado con `validateDocumentForPersist` — rechaza vaciar Spec con contenido sustancial, shell solo-changelog o reducción peligrosa (>45% del cuerpo previo).
- **SDD / spec-kit:** `GET /projects/:id/export/sdd-bundle` (JSON layout spec-kit); `GET /projects/:id/export/repo-handoff` (spec-kit + gobernanza reconciliada o sintetizada + espejos `docs/sdd/` + `openspec/BRANCH-POLICY.md`); `POST /projects/:id/clarify-spec`; `GET /projects/:id/analyze?stageId=`; `GET /projects/:id/next-task`; `POST /projects/:id/converge?stageId=`; `POST /projects/:id/tasks-to-issues` (`GITHUB_TOKEN`). Ver `sdd-integration.service.ts`, `handoff-export.util.ts`, `docs/plans/PLAN-SPECKIT-ALIGNMENT.md` y `docs/speckit-vs-theforge.md`.
- **Hermes:** `POST /projects/:id/launch-hermes` envía `sddBundle` (spec-kit-compatible) + `implementHandoff` en el webhook.
- **Re-valorar complejidad:** `POST /projects/:id/reassess-complexity` (body opcional `{ note?: string }`) — vuelve a inferir `complexityPending` desde DBGA/MDD/Spec existentes sin re-ejecutar el stream DBGA.
- **BRD greenfield:** `POST /projects/:id/suggest-brd-from-dbga` (body opcional `{ stageId?: string }`) — borrador BRD desde `Project.dbgaContent` (mín. ~300 caracteres); rechaza `projectType === LEGACY` (usar `POST …/legacy/suggest-brd-from-codebase-doc`). Prompt `brd-generation-prompt`: negocio puro (capacidades, UAT, entidades, políticas comerciales); **§4 diagramas Mermaid** (ecosistema, ER de negocio, 2–3 flujos); traduce lo técnico del DBGA; sin endpoints ni SQL.
- **AEM:** `POST /projects/:id/generate-aem` (body `{ marketScope: "global" \| "mexico" \| "latam" }`) — Análisis y Estudio de Mercado desde Benchmark (Deep Research), Fase 0 (DBGA) y BRD; segundo paso añade **Dictamen de inversión digital** (SEGUIR / NO SEGUIR / SEGUIR CON CONDICIONES). Persistido en `aemContent`. Prompts `aem-prompt.md` + `aem-investment-advisor-prompt.md`.
