# Sessions

Gestión de sesiones de chat del Workshop y parseo de respuestas que incluyen documentos (MDD, UX/UI, DBGA, BRD por etapa, Manual To-Be, etc.).

- **SessionsService:** CRUD de sesiones, `chat`, `chatStream`, `generateWelcome`. `chat` / `chatStream` los usa **`AiOrchestratorService`** (persistencia del log y llamada al modelo); no son el contrato HTTP de `POST :id/chat`. Opciones LLM: `currentBrdContent` / `currentToBeManualContent` cuando `activeTab` es `brd` o `to-be`; el `done` puede devolver `brdContent` / `toBeManualContent` parseados con `---FIN_BRD---` y `---FIN_TOBE---`. En tab **ux-ui-guide**, reenvía también `uxGuideDesignRef*` (bloque de referencia visual del Design System) hacia `AiService`. **`generateWelcome`:** pasa `activeTab` y **`welcomeBrief: true`** al `AiService` (system corto en lugar de `MASTER_PROMPT` + sin inyectar documentos enteros en system; el contexto acotado va en el **user** `syntheticPrompt`). Ante **429** del proveedor (p. ej. OpenRouter) puede aplicarse reintento con backoff según orquestación; en rate limit sostenido puede persistirse un **mensaje estático** de bienvenida (evita 500 en producción). Mensajes en `chatLog` pueden incluir **`stageId`**. `POST :id/messages` acepta `stageId` opcional (`appendChatSchema`). **Imágenes (hasta 6):** `describeImagesForChat` usa el **modelo de visión** de la instancia activa (`createForVisionUser`); el bloque `VISION_CONTEXT_HEADER` queda en `content` (las `images` en el log son solo UI). El **stream de chat** envía al LLM solo texto (`sessionHistoryToLlm` sin binarios) para no invocar el modelo de chat con `image_url` (p. ej. deepseek → 404). **Intención (router híbrido):** `IntentRouterService.route` clasifica cada turno (`chat_only` / `edit_document` / `confirm_then_edit`); heurística primero, LLM solo si ambiguo (`INTENT_ROUTER_LLM=0` desactiva el paso LLM). **Ediciones DBGA (tab `benchmark`):** si `action === edit_document`, **`tryBenchmarkDbgaEditTurn`** ejecuta solo `BENCHMARK_REFINE` (incl. spec pegada vía heurística o LLM). Brainstorming explícito (`isUserExploringDbgaIntent`) → `chat_only`. Preguntas del asistente tipo «¿incorpore al DBGA?» no disparan aviso de panel sin persistir (`isHypotheticalDocumentEditOffer`).
- **`POST :id/chat` (SessionsController):** delega en **`AiOrchestratorService.chatBySessionId`** — misma lógica que **`POST /ai-orchestrator/chat`** (HITL complejidad, PATCH MDD/UX/DBGA desde body, supervisor, legacy/TheForge, **`uxGuideLlmOptions`** cuando toca, ingest SDD, evaluador). Body: `message` y/o **`images`** (`parseChatImageAttachments`), opcionales `activeTab`, `stageId`, `mddContent`, `uxUiGuideContent`, `dbgaContent`. Respuesta: **`{ session, project, uxUiGuideContent?, evaluatorCritique? }`** (no el retorno plano interno de `SessionsService.chat`).
- **ChatResponseParserService:** Separa documento vs mensaje de chat (delimitadores `---FIN_MDD---`, `---FIN_DBGA---`, etc.), fusiona secciones de MDD/DBGA y fallback cuando el modelo omite el delimitador (incl. Fase 0 / tab `benchmark`, títulos tipo *Research Report*, outline numerado, intro «He integrado…» + cuerpo). **`looksLikeApiEndpointCatalog`:** listas GET/POST no sustituyen el panel (se anexan). `SessionsService.salvageDbgaFromAssistantText` recupera DBGA pegado solo en el chat. Si el modelo responde solo «He actualizado…» sin `---FIN_DBGA---`, `refineDbgaFromUserRequest` relanza el refinado con `BENCHMARK_REFINE` y el DBGA actual. Con **imágenes**, la descripción de visión va al refinado (no solo el texto corto del usuario). `benchmarkAssistantChatMessage` colapsa el cuerpo del documento en el chat cuando sí se persistió (`looksLikeDbgaDocumentBody`). Tab **benchmark** + sub-tab Fase 0 edita `dbgaContent`; el Benchmark (Deep Research) es `phase0SummaryContent` (no se pisan entre sí).
- **document-content.util.ts:** Funciones puras `normalizeDashes`, `stripChatLabel`, `cleanDocumentContent` (formatea markdown y garantiza `## Registro de cambios del documento` con fila 1.0 si falta). Sin DI.
- **document-shrink.util.ts:** `wouldShrinkDocDangerously`, `validateDocumentForPersist`, `documentPersistFieldLabel` — evita persistir documentos vacíos, solo-changelog o que borren la mayor parte del contenido previo (chat orquestador, `clarify-spec`, `PATCH specContent`, `PATCH dbgaContent`, `legacy/generate-from-codebase`).
- **`POST project/:projectId/salvage-dbga`:** recupera DBGA del chat benchmark; snapshot del estado actual antes de restaurar. Ver también **`GET/POST …/document-snapshots`** en Projects.
- **orchestrator-doc-guard.util.ts:** Heurísticas compartidas para tabs del orquestador (Arquitectura, Spec, Infra, etc.): si el usuario pidió cambio o el asistente afirma haber editado pero no hubo `---FIN_TAG---` / parseo, `SessionsService` añade aviso en el mensaje del chat y el frontend muestra error si el documento del panel no cambió.
