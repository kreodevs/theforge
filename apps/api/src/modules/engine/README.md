# Engine (SDD)

- **`mdd-markdown-parser.ts`:** `normalizeMddContent` / `parseMarkdownMdd` producen JSON compatible con `mddJsonSchema` (incluye **`constitution`**: señales «Constitución Cursor» desde §1–§2–§5 para el semáforo HIGH cuando Fase B fusione puertas con `sddDomainGraphOk`). Tests: `mdd-markdown-parser.constitution.spec.ts`.
- **`semaphore.service.ts`:** calcula `Stage.status` / `precisionScore` según `Project.complexity` (`ComplexityLevel`): **LOW** (HU + tareas), **MEDIUM** (spec o UC, API, UX o flujos, **historias de usuario**, tareas — 5 gates; 5/5 VERDE, ≥3 AMARILLO), **HIGH** (MDD JSON + Figma si `hasUxTeam`). En HIGH, si `sddDomainGraphOk` (Grafo Falkor sin dependencias huérfanas §3–§4), puede pasar a VERDE con `precisionScore` 92 aunque falten textos largos de `edge_cases`/`field_types`. Si `mddJson.constitution.template_detected` (Constitución Cursor), puertas extra (mapa, glosario, Gherkin, bloqueantes, ADR/stack): incumplimiento → AMARILLO; se fusiona con el resultado base (un VERDE puede bajar; un AMARILLO solo si la constitución es peor — score más bajo).
- **`mdd-update-pipeline.service.ts`:** valida/sanitiza MDD, sincroniza el borrador al Grafo SDD (`GraphMemoryService`) y evalúa `evaluateSddDependencyHealth` antes del semáforo cuando hay `projectId`/`stageId` y complejidad HIGH.
- **Costes / conformidad:** `cost-calculator.service.ts` delega la fórmula en `@theforge/business-rules`; `conformance.service.ts`; reparación post-generación Blueprint en `blueprint-conformance-repair.util.ts`; Contratos API en `api-conformance-repair.util.ts` (normalización `{id}`/`:id`, filas §4 faltantes).
